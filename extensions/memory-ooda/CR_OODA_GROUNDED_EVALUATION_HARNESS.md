# CR: OODA Grounded Evaluation Harness — Measurable Outcome Metrics

**Date:** 2026-04-17
**Status:** WRITTEN
**Priority:** CRITICAL — without this, every adaptive mechanism (weight proposals, trajectory scaling, meta-reviewer) optimizes against LLM-generated labels, not ground truth
**Author:** Design session with michaelpeter
**Source:** DeepMind "Discovering Multiagent Learning Algorithms with LLMs" (2602.16928v2) — AlphaEvolve's fixed evaluation harness principle

---

## Problem

AlphaEvolve's key architectural insight: the **evolvable skeleton** (regret accumulation, policy derivation) mutates freely, but the **evaluation harness** (exploitability computation) is exact and unchanging. This separation is what makes the evolutionary search trustworthy — the algorithm can't game its own fitness function.

OODA has an evolvable skeleton (triage → strategy → council → act) and a feedback loop (outcome labeling → archivist → valuation engine → weight proposals → meta-reviewer). But the evaluation harness — the thing that measures whether the system is actually getting better — is missing.

Current state of outcome measurement:

1. **Outcome labeling (O3 in archivist.ts):** Labels are `"success" | "failure" | "partial"` stored on episodic events. These are either inferred by the LLM during the archivist's distillation pass or derived from `outcomeSignal` strings. There is no objective measurement — the agent grades its own homework.

2. **Domain outcome aggregation (V1 in archivist.ts):** `aggregateDomainOutcomes()` counts successes/failures per domain and computes a success rate. This feeds `shouldProposeWeightAdjustment()` which proposes domain weight changes when `|successRate - currentWeight| > 0.2`. But the success rate is computed from self-assessed labels.

3. **Weight proposals (V3):** Generated by the archivist, reviewed by the meta-reviewer. Both rely on the same LLM-generated outcome data. If the labels are wrong, the proposals are wrong, and the weight adjustments drift.

4. **CR_OODA_TRAJECTORY_AWARE_TRIAGE** (companion CR): Introduces trajectory-based signal scaling in triage. The trajectory is computed from outcome labels. If the labels are unreliable, trajectory scaling amplifies noise instead of signal.

The net effect: the system has a feedback loop, but the feedback signal is ungrounded. This is equivalent to AlphaEvolve letting the evolved algorithm also write its own exploitability score.

## Design

### Grounded Metrics per Domain

Define a measurable, non-LLM-assessed metric for each registered domain. These metrics are computed from observable system state, not from model inference:

| Domain           | Metric                   | Source                                       | Computation                                                           |
| ---------------- | ------------------------ | -------------------------------------------- | --------------------------------------------------------------------- |
| `openclooda`     | Archivist health score   | `ooda-health.json` + `.archivist-state.json` | `(completions / scheduled) × (1 - error_rate)` over last 7 days       |
| `amf_pipeline`   | Parity score             | Pipeline API / gauntlet output               | Latest `parity` field from `lastRun` in world model project state     |
| `infrastructure` | Gateway uptime           | launchd + health-monitor                     | `(uptime_seconds / window_seconds)` from health-monitor pings         |
| `testing`        | Test pass rate           | Vitest output / CI                           | `(passed / total)` from last test run                                 |
| `unknown`        | Fallback: LLM label rate | Episodic store                               | `successes / total` — keeps current behavior for unregistered domains |

Each domain's metric resolver is a pure function: `(domainId, episodicStore, systemState) → number` in [0.0, 1.0]. The metric is the ground truth that weight proposals and trajectory computation are measured against.

### Metric Registry

```typescript
export interface DomainMetric {
  domainId: string;
  /** Compute the objective metric for this domain. Returns [0.0, 1.0]. */
  compute: (ctx: MetricContext) => Promise<number>;
  /** Human-readable description of what this metric measures. */
  description: string;
}

export interface MetricContext {
  workspacePath: string;
  episodicStore: EpisodicStore;
  worldModelStore: WorldModelStore;
  healthPath: string;
}
```

Domain metrics are registered in a `metric-registry.ts` module. New domains can add their metric resolver without touching the core OODA chain. Unregistered domains fall back to the current LLM-label-based success rate.

### Integration Points

1. **Outcome labeling (O3):** After the archivist's LLM-based labeling pass, cross-check against the grounded metric. If the grounded metric for a domain is > 0.7 but the LLM labeled recent events as "failure," flag the discrepancy in the archivist log. Don't override the LLM label — record both and let the meta-reviewer adjudicate.

2. **Weight proposals (V1/V3):** `aggregateDomainOutcomes()` currently uses LLM labels exclusively. Add a `groundedScore` field to `DomainOutcomeStats` populated from the metric registry. `shouldProposeWeightAdjustment()` uses `groundedScore` when available, falls back to `successRate` for unregistered domains.

3. **Trajectory computation (C3 of CR_OODA_TRAJECTORY_AWARE_TRIAGE):** When computing domain trajectories, prefer the grounded metric's time series over LLM label counts. For domains with a registered metric, trajectory = `(current_metric - metric_30d_ago) / metric_30d_ago`. For unregistered domains, fall back to label-based trajectory.

4. **Meta-reviewer:** When reviewing weight proposals, the meta-reviewer receives both the LLM-label-based stats and the grounded metric for each domain. This is the audit surface — the meta-reviewer can detect when the LLM's self-assessment diverges from measured reality.

## Changes Required

### C1 — `metric-registry.ts` (NEW FILE)

```typescript
export interface DomainMetric { ... }
export interface MetricContext { ... }
export interface MetricResult {
  domainId: string;
  score: number;          // [0.0, 1.0]
  description: string;
  computedAt: number;
}

/** Registry of grounded domain metrics. */
export class MetricRegistry {
  private metrics = new Map<string, DomainMetric>();

  register(metric: DomainMetric): void;

  /** Compute metric for a domain. Returns null for unregistered domains. */
  async compute(domainId: string, ctx: MetricContext): Promise<MetricResult | null>;

  /** Compute all registered metrics. */
  async computeAll(ctx: MetricContext): Promise<MetricResult[]>;
}
```

Built-in metric resolvers for `openclooda`, `amf_pipeline`, `infrastructure`, `testing` as described in the table above.

### C2 — `types.ts` (MODIFY)

Add to `DomainOutcomeStats`:

```typescript
export interface DomainOutcomeStats {
  domain: string;
  decisions: number;
  successes: number;
  failures: number;
  partials: number;
  successRate: number;
  /** Grounded metric from MetricRegistry, if available. [0.0, 1.0]. */
  groundedScore?: number;
  /** Description of the grounded metric source. */
  groundedMetricSource?: string;
}
```

### C3 — `archivist.ts` (MODIFY)

In `runArchivist`, after step 3c (outcome stats):

1. Instantiate `MetricRegistry` with built-in resolvers
2. Call `registry.computeAll(ctx)` to get grounded scores
3. Attach `groundedScore` to each `DomainOutcomeStats` entry
4. In the archivist log, record grounded vs. LLM-label discrepancies when `|groundedScore - successRate| > 0.3`

In `shouldProposeWeightAdjustment()`:

1. If `stats.groundedScore` is defined, use it instead of `stats.successRate` for the delta computation
2. Log which metric was used in the proposal rationale

### C4 — `meta-reviewer.ts` (MODIFY)

When building the meta-reviewer prompt, include both:

- LLM-label stats: `successRate`, outcome counts
- Grounded metric: `groundedScore`, metric source description

Add an explicit instruction: "If the grounded metric and LLM-label success rate disagree by more than 30%, flag this as a calibration issue and do NOT approve weight changes based on the LLM labels alone."

### C5 — `index.ts` (MODIFY)

During plugin registration, instantiate the `MetricRegistry` with built-in resolvers. Pass it to the archivist runner via `ArchivistRunnerDeps`.

### C6 — Tests

- `metric-registry.test.ts`: test each built-in resolver with mock system state
- `archivist.test.ts`: test that `groundedScore` is attached to outcome stats and that discrepancy logging fires
- `valuation-engine.test.ts`: test that `shouldProposeWeightAdjustment` prefers grounded score when available

## Phasing

**Phase 1 (this CR):** MetricRegistry with `openclooda` and `amf_pipeline` resolvers + archivist integration. These two domains have clear, measurable metrics available today.

**Phase 2 (follow-up):** Add `infrastructure` and `testing` resolvers. Integrate with trajectory computation from CR_OODA_TRAJECTORY_AWARE_TRIAGE. Build the meta-reviewer calibration audit surface.

## Non-Goals

- This CR does NOT replace LLM-based outcome labeling. The grounded metric is a parallel signal for cross-validation, not a replacement. LLM labels capture nuance (partial success, context) that a scalar metric can't. The goal is to prevent the feedback loop from drifting when the LLM's self-assessment diverges from reality.
- This CR does NOT introduce automated weight adjustment. Weight proposals still require meta-reviewer approval. The grounded metric makes that approval decision more informed, not automatic.
