# CR_OODA_ROADMAP_TIER5_META_REVIEWER

**Priority:** P2  
**Date:** 2026-04-05  
**Status:** WRITTEN  
**Tier:** 5 — Meta-Reviewer  
**Depends on:** Tier 2 (outcomes), Tier 3 (valuation), Tier 4 (cross-project)  
**Goal:** Periodic autonomous review of agent behavior against observed outcomes. Proposes specific, actionable changes to prompts, weights, and strategies. Human approves all changes.

---

## Problem

The Proposals system (P5) surfaces issues reactively — during a session when the Archivist notices a pattern. But there's no systematic review that runs independently of active sessions, compares current behavior against outcomes longitudinally, and identifies structural drift.

Example gaps:

- The Triage prompt hasn't changed since it was written. Is it still accurate given 3 weeks of sessions?
- The Strategy model consistently selects "low-risk" strategies that fail. Is the risk axis calibrated correctly?
- The Archivist's `extractProposals` generates proposals about X but not Y — is Y being missed systematically?

A Meta-Reviewer runs on a schedule, not in-session, and produces a structured diagnostic report that humans review.

---

## Design

### M1 — Meta-Reviewer cron job

Schedule: weekly (Sunday 23:00 or on-demand via CLI).

**New file:** `scripts/meta-review.ts`

Inputs:

- Last 30 days of episodic memories with outcome labels (Tier 2)
- KNOWLEDGE.json current state
- PRIORITIES.json current weights
- Current triage.ts / strategy.ts prompt text
- Proposal history (approved/rejected)

Outputs: `meta-review/YYYY-MM-DD.md`

### M2 — Meta-Reviewer analysis passes

**Pass 1: Outcome audit**

```
For each domain (amf_pipeline, openclooda, dreamboard, iep_champion):
  - Total decisions, success rate, trend (improving/declining)
  - Top 3 failure modes with example memories
  - Recommendation: no change | increase/decrease weight | modify strategy
```

**Pass 2: Prompt effectiveness**
Compare what triage/strategy generate vs what actually gets executed:

- If Strategy consistently proposes Option A but user always overrides to Option B → prompt is miscalibrated for this domain
- If Triage priority scores don't correlate with time-to-resolution → scoring criteria need adjustment

**Pass 3: Knowledge gap detection**
Run a coverage check: are there active project areas with <3 KNOWLEDGE.json entries? Are there recurring topics in episodic memory that haven't been promoted to KNOWLEDGE.json?

**Pass 4: Proposal effectiveness**
Of proposals generated by P5:

- How many were approved vs rejected?
- Of approved: did they improve outcomes?
- Of rejected: why? (no reason stored = deferred M6 item)

### M3 — Meta-Reviewer output format

```markdown
# Meta-Review 2026-04-05

## Outcome Audit

### amf_pipeline

- 23 decisions, 61% success rate (↑ from 48% last month)
- Top failure mode: module scoping (4 incidents) — covered by lessons_learned
- Recommendation: no weight change; weight=0.9 appropriate given active client work

### openclooda

- 8 decisions, 87% success rate
- All failures were test-related (intermittent)
- Recommendation: no change

## Prompt Effectiveness

- Triage: priority scores correlate with actual urgency (r=0.73) — healthy
- Strategy: Option "write CR first, implement after" selected 6/8 times and succeeded 5/6 — good signal
- ANOMALY: User overrode Strategy recommendation 3 times for AMF runs — Strategy may be too conservative on P0 bugs

## Knowledge Gaps

- IEP Champion: 2 KNOWLEDGE.json entries, 12 recent episodic memories — should promote
- Dreamboard: 0 KNOWLEDGE.json entries about the Joanna pilot — add

## Proposals Effectiveness

- 5 proposals generated, 3 approved, 2 ignored
- Approved proposals improved subsequent parity scores in 2/3 cases
- No outcome data for rejected proposals (M6 gap)

## Recommended Actions

1. [ ] Promote IEP Champion episodic memories to KNOWLEDGE.json (3 entries)
2. [ ] Review Strategy conservatism on P0 AMF bugs — consider separate AMF strategy mode
3. [ ] Add Dreamboard Joanna pilot context to KNOWLEDGE.json
```

### M4 — Human review flow

Meta-review output is:

1. Written to `meta-review/YYYY-MM-DD.md`
2. Announced via cron delivery to the main session
3. Actionable items are optionally converted to Proposals (M-numbered)

No autonomous changes. The Meta-Reviewer observes and recommends. The user decides.

### M5 — After_tool hook for prompt mutation tracking

If future work allows prompt text to be modified (e.g., editing `triage.ts` prompt string), the Meta-Reviewer needs to track when prompts changed and correlate with outcome shifts before/after. Hook: when `write` or `edit` tool touches `extensions/memory-ooda/*.ts`, capture a `prompt_mutation` structural event (already handled by C3's after_tool_call handler).

## Open Questions

- **Prompt effectiveness measurement:** Correlating Strategy recommendations with actual user action requires tracking what the user did after seeing a Strategy, not just what Strategy proposed. This needs turn-level tracking of "strategy seen → action taken" pairs. May need a lightweight feedback signal (user explicitly says "following your suggestion" or tool calls match strategy recommendation).

- **Proposal rejection reasons:** Currently proposals are approved or ignored — no explicit rejection with reason. A rejected proposal can't be learned from. This is M6 from the peer review. Implement a `dismissProposal(id, reason)` flow before Tier 5 is fully useful.
