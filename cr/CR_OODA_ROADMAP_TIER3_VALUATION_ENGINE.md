# CR_OODA_ROADMAP_TIER3_VALUATION_ENGINE

**Priority:** P1  
**Date:** 2026-04-05  
**Status:** WRITTEN  
**Tier:** 3 — Valuation Engine (closes the weight update loop)  
**Depends on:** Tier 2 (outcome labeling)  
**Goal:** Domain weights in PRIORITIES.json currently set by hand and never updated. The Valuation Engine should auto-propose weight adjustments based on observed outcome patterns. Not autonomous — proposes, human approves.

---

## Problem

`PRIORITIES.json` has `domains` with weights (0.1–1.0) and a `scoring_rubric`. These are set once and never change. The valuation engine scores strategies but never feeds back into the weights. If the agent consistently makes bad decisions on AMF pipeline work (low parity scores) but good decisions on OpenCLOODA work, the weights should reflect that — but they don't.

The Proposals system (P5 CR) surfaces proposals reactively. The Valuation Engine should drive weight proposals proactively, from observed outcomes.

---

## Design

### V1 — Outcome aggregation per domain

After Tier 2 is in place, the Archivist can aggregate outcome data per domain:

```typescript
interface DomainOutcomeStats {
  domain: string; // "amf_pipeline", "openclooda", etc.
  decisions: number; // total actionId decisions in last 30 days
  successes: number;
  failures: number;
  partials: number;
  successRate: number; // successes / decisions
}
```

Domain is inferred from memory text (keyword matching: "amf", "pipeline", "kohlscore" → amf_pipeline; "ooda", "archivist", "triage" → openclooda; etc.).

### V2 — Weight proposal generation

If a domain's success rate deviates significantly from its weight (e.g., amf_pipeline has weight=0.9 but 60% failure rate), generate a weight proposal:

```typescript
function shouldProposeWeightAdjustment(
  stats: DomainOutcomeStats,
  currentWeight: number,
): WeightProposal | null {
  const expectedSuccessRate = currentWeight; // rough proxy
  const delta = stats.successRate - expectedSuccessRate;
  if (Math.abs(delta) > 0.2 && stats.decisions >= 5) {
    const proposedWeight = clamp(currentWeight + delta * 0.3, 0.1, 1.0);
    return {
      domain: stats.domain,
      currentWeight,
      proposedWeight,
      rationale: `${stats.decisions} decisions, ${(stats.successRate * 100).toFixed(0)}% success rate vs ${(currentWeight * 100).toFixed(0)}% weight`,
    };
  }
  return null;
}
```

Adjustment is conservative (0.3× of delta), requires ≥5 decisions for statistical basis.

### V3 — Integration with Proposals system

Weight proposals feed into the existing `PolicyProposal` system (P5):

```typescript
// In extractProposals (archivist.ts)
const weightProposals = generateWeightProposals(domainStats, currentPriorities);
for (const wp of weightProposals) {
  await addProposal({
    id: `weight-${wp.domain}-${Date.now()}`,
    category: "weight_adjustment",
    description: `Adjust ${wp.domain} weight: ${wp.currentWeight} → ${wp.proposedWeight}`,
    rationale: wp.rationale,
    confidence: Math.min(0.9, stats.decisions / 20),
    autoGenerated: true,
  });
}
```

User approves via the existing proposal review flow. On approval: `PRIORITIES.json` is updated, meta-reviewer notes the change.

### V4 — Scoring rubric calibration

The scoring rubric (alignment/efficiency/risk weights) should also be observable. Track which axis most often caused a strategy to be selected that later failed:

- If high-alignment strategies consistently fail → alignment weight may be over-indexed
- If low-risk strategies consistently succeed when higher-risk ones failed → risk weight validated

Surface this as a diagnostic in the SITREP attention field when ≥3 data points exist:

```
**ATTENTION:** Risk weighting appears under-indexed — 4/5 recent failures were high-alignment, high-risk strategies
```
